# Contributing

这份文档说明如何继续往这个知识库里增加内容，或者升级已有内容。

这里的“贡献”不是提交代码功能，而是继续产出高质量的 Markdown 知识文档。  
目标不是把仓库写大，而是把每篇文档写到以后还能反复调用的程度。

## 贡献什么

这个仓库主要接受三类内容：

- 新概念文档
- 旧文档升级
- 方法论文档改进

对应地，常见工作只有三种：

1. 为一个新概念写一篇完整文档
2. 按统一规范升级一篇旧文档
3. 优化生成规范、模板或方法论

## 先理解这个仓库的目标

在开始写之前，先确认你理解这个仓库要的不是哪一类文档。

这里不要的是：

- 术语解释页
- 百科式摘要
- 东拼西凑的资料摘抄
- 只有定义没有机制的笔记
- 只有教程步骤没有模型的说明

这里要的是：

- 能解释问题定义的文档
- 能说清边界的文档
- 能给出核心结构和因果链的文档
- 能迁移到相邻问题的文档
- 能帮助以后分析问题的文档

一句话说，贡献的标准不是“写完了”，而是“以后还能拿出来用”。

## 新文档放哪里

所有正式知识文档都放在 `docs/` 下，按主题分目录。

当前已有主题包括：

- `docs/ai-systems/`
- `docs/computer-systems/`
- `docs/programming-languages/`
- `docs/image-processing/`
- `docs/methodology/`

主题选择原则：

- 概念主要属于哪个问题域，就放到哪个主题
- 如果一个概念跨多层，优先放到最能承载其主问题的主题
- 不要因为实现语言不同，就把系统概念误放进语言主题
- 不要为了整齐新建大量细碎目录

如果现有主题都不合适，再考虑新增主题目录。

## 文件命名规则

- 文件名使用 `kebab-case`
- 名称尽量直接对应概念本身
- 如果文档强调“学习模型”或“推导”，可以在文件名中显式体现
- 不要用过于口语化、临时性或模糊的文件名

示例：

- `memory-order.md`
- `virtual-memory-learning-model.md`
- `guided-filter-derivation.md`

## 新建文档的标准流程

### 1. 明确最小输入

开始之前，至少明确下面四项：

- 概念名
- 所属主题
- 你是在什么语境下遇到它的
- 你当前最不理解的点

如果这四项都说不清，文档通常会写散。

### 2. 判断是“新建”还是“升级”

- 仓库里还没有对应文档：新建
- 仓库里已经有对应文档，但结构不完整或质量不够：升级

不要为了省事，重复造一篇几乎相同的新文档。

### 3. 尽量使用固定生成指令

最短可用指令：

```text
按 docs/methodology/concept-document-template.md 和两份 playbook，为概念 {concept} 生成一篇新的知识文档，并直接放入合适的 docs/{topic}/ 目录。不要写成泛泛解释，必须包含：问题定义、对象边界、核心结构、机制/因果链、关键 tradeoff 与失败模式、应用场景、工业/现实世界锚点、当前推荐实践、过时路径与替代、自测题、迁移入口、未解问题、参考资料，并补齐统一 frontmatter。
```

升级旧文档的指令：

```text
按 docs/methodology/concept-document-template.md 升级现有文档 {path}。

要求：
- 保留原有高价值内容，不要整篇重写
- 补齐当前缺失的标准章节和 frontmatter 字段
- 把文档提升到可保留、可复用、可迁移的状态
- 重点补：工业/现实世界锚点、当前推荐实践、过时路径与替代、自测题、迁移入口、未解问题
```

### 4. 生成后做结构检查

新文档至少应覆盖这些信息位点：

1. 这份文档要帮你学会什么
2. 一句话结论 / 问题定义
3. 对象边界与相邻概念
4. 核心结构
5. 核心机制 / 主链路 / 因果链
6. 关键 tradeoff 与失败模式
7. 应用场景
8. 工业 / 现实世界锚点
9. 当前推荐实践、过时路径与替代
10. 自测题 / 验证入口
11. 迁移与关联模型
12. 未解问题与继续深挖
13. 参考资料

不是每篇都必须逐字使用同样标题，但这些信息不能缺。

### 5. 补齐 frontmatter

frontmatter 至少包含：

- `doc_id`
- `title`
- `concept`
- `topic`
- `created_at`
- `updated_at`
- `source_basis`
- `time_context`
- `applicability`
- `prompt_version`
- `template_version`
- `quality_status`
- `related_docs`
- `open_questions`

### 6. 更新导航

只要新增了正式文档，就同步更新 [docs/index.md](/Users/maxwell/Knowledge/docs/index.md)。

## 写作规则

### 以模型为中心，不以术语为中心

文档要让读者获得的是内部模型，而不是更多定义。

优先回答：

- 它在解决什么问题
- 为什么需要它
- 它依赖哪些关键结构
- 它如何运转
- 何时成立，何时失效

### 必须有边界

如果一个概念容易和相邻概念混淆，必须主动拆边界。  
例如：

- 它不等于什么
- 它和哪些概念经常被混用
- 它只解决哪一层问题

### 必须有现实锚点

工业或现实世界锚点不能是空话。

不够好的写法：

- “在工业界很常见”
- “很多系统都会用到”

更好的写法：

- 它在哪类系统里被频繁使用
- 工程上为什么关心它
- 如果理解错，会出什么问题

### 必须有过时路径与替代

如果文档提到旧方案、旧技术或被淘汰路径，必须解释：

- 它为什么旧
- 局限在哪里
- 现在更推荐什么

不能只写一句“这个已经不推荐了”。

### 必须有验证入口

每篇文档都应至少包含一种验证方式：

- 自测题
- 纠错题
- 预测题
- 反例检查

否则很容易停留在“读的时候像懂了”。

### 必须有迁移入口

文档最后应能把模型引向相邻问题。  
如果一个概念学完后完全无法迁移，那篇文档通常还不够成熟。

## 来源与事实要求

### 时间敏感内容必须显式带日期

这些内容默认都算时间敏感：

- 标准状态
- 语言特性变化
- 库行为
- 工具链行为
- 官方推荐实践
- 法规、规则、产品规格、版本现状

写法上必须明确日期，例如：

- “截至 2026-03-16”
- “按 2026-03-16 核对的当前工作草案”

### 规范性内容优先使用一手资料

优先使用：

- 标准文本
- 官方文档
- 论文原文
- 官方技术说明

尽量不要把二手博客当成规范性结论的主要依据。

### 不确定时先核对，不要凭印象写

如果一个事实有明显时效性，或者你对它没有足够把握，先核对再写。  
知识库最怕的不是空白，而是过期但看起来很像真的结论。

## 升级旧文档的原则

升级旧文档时，默认原则是“保留高价值内容，补齐缺口”，而不是推倒重来。

优先补下面几类缺口：

- 缺少问题定义
- 缺少对象边界
- 缺少机制解释
- 缺少现实锚点
- 缺少当前推荐实践
- 缺少过时路径与替代
- 缺少验证入口
- 缺少迁移入口
- 缺少未解问题

## 不要做的事

- 不要新增重复文档来覆盖已有主题
- 不要只写定义和例子就结束
- 不要用空泛案例代替真实应用场景
- 不要省略时间语境
- 不要引用过时做法却不解释替代路径
- 不要为了“看起来整齐”删掉关键专业细节
- 不要把文档写成“我当时读资料时的过程流水账”

## 最后检查清单

在认为文档完成之前，至少检查一遍：

- 目录位置是否合理
- 文件名是否符合 `kebab-case`
- frontmatter 是否齐全
- 是否真的回答了“问题-边界-结构-机制-失效-应用-替代”
- 是否包含自测或验证入口
- 是否包含迁移入口
- 是否包含未解问题
- 如果有时间敏感结论，是否写清日期
- 如果是新文档，是否更新了 [docs/index.md](/Users/maxwell/Knowledge/docs/index.md)

## 推荐起手方式

如果你只是想稳定地产出新文档，最简单的习惯是：

1. 先在脑子里明确概念、主题、语境、困惑
2. 直接使用固定生成指令
3. 生成后对照模板和本文件做一次补漏
4. 更新 `docs/index.md`

这比自由发挥更稳，也更符合这个仓库的定位。
