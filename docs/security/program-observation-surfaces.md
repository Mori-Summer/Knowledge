# 未授权程序观测逻辑：把 DMA、进程内存与抓包放回同一状态获取模型

## 1. 这份文档要帮你学会什么

这篇文档的目标不是解释几个零散术语，而是把“DMA、内存、抓包为什么都能进入外挂类程序的观察链”压成一套可以反复调用的内部模型。

读完后，你应该至少能做到：

- 不再把这三类路径看成互不相关的技巧，而是看成三种不同的**状态投影面**
- 解释为什么同样叫“拿信息”，DMA 拿到的是原始页级投影，进程内存拿到的是高语义快照，抓包拿到的是协议序列化投影
- 判断一类未授权观测为什么偏好某个面，以及它为什么又常常需要把多个面拼起来
- 把问题从“封不封某个 API”升级成“哪些高价值状态在客户端上可见、多稳定、多容易被重建”
- 把这套模型迁移到 anti-cheat、客户端安全架构、协议设计和威胁建模中

本文只讨论**防御视角下的建模与判断框架**。  
不会给出未授权实现步骤、绕过路径或可直接复用的操作配方。

## 2. 一句话结论 / 问题定义

**外挂类程序之所以能工作，通常不是因为它们“直接看到了程序的真相”，而是因为它们从 DMA、进程内存或网络抓包这些不同观测面上拿到程序状态的投影，再通过解码、时间对齐和规则推断，把投影重建成足够可用的高层状态。**

真正要解决的问题不是“能不能读到一点数据”，而是下面五件事是否同时成立：

- 是否存在可达的观测面
- 观测面里是否真的承载了高价值状态
- 这些状态是否足够稳定，能支持重复采样和时间对齐
- 观测得到的是原始字节、对象快照，还是协议事件
- 防御方是否把真正的裁决权仍然留在客户端之外

可以把未授权程序观测压成一个通用公式：

**可用状态 = 可见投影 × 解码能力 × 时间对齐 × 置信度管理**

防御的核心也因此不是“永远不让别人看到任何字节”，而是：

- 减少客户端长期持有的高价值真值
- 增加投影与真值之间的距离
- 把真正不可替代的裁决权收回到更难观测的位置

## 3. 对象边界与相邻概念

### 3.1 这篇文档直接处理什么

这篇文档直接处理的是：

- DMA、进程内存读取、网络抓包这三类观测面分别能看到什么
- 为什么这些观测面会形成外挂类工具常见的状态获取链
- 对防御者来说，应该在哪些层收缩暴露面，而不是只盯具体工具名

### 3.2 它不等于什么

它不等于：

- **外挂开发教程。**  
  本文不讨论实现路径、代码模板和绕过手法。

- **单一操作系统 API 说明。**  
  API 只是观测面的入口之一，不是模型本体。

- **“只要加密就没事”的网络安全短句。**  
  加密只能收窄某一面，不会自动消灭其他可见面。

- **“只要客户端做得花就没人能看懂”的反作弊神话。**  
  外观混淆不等于状态不可重建。

### 3.4 先把三种“可见性”拆开

最容易混淆的不是“它们都能看到数据”，而是三种完全不同的可见性：

- **物理或总线可见性。**  
  DMA 看到的是某些页框或 IOVA 背后的原始内容。

- **地址空间可见性。**  
  进程内存观测看到的是运行时对象、缓存、队列和布局后的程序状态。

- **协议可见性。**  
  抓包看到的是程序决定发到线上去的消息、重传、握手、时序和方向。

这三种可见性不在一个语义层上。  
把它们混为一谈，就会得出很多错误防御结论。

## 4. 核心结构

### 4.1 最稳的统一模型：六个平面

理解这类问题，最稳的方式不是按工具名分类，而是按六个平面分类：

| 平面 | 你要问什么 | DMA | 进程内存 | 抓包 |
| --- | --- | --- | --- | --- |
| 采集平面 | 数据从哪里被拿出来 | 总线 / 设备侧 | 进程地址空间 | 网络接口 / 流 |
| 状态承载平面 | 里面装的是什么 | 原始页内容 | 对象、缓存、布局后的状态 | 序列化消息与时序 |
| 语义密度平面 | 离高层语义有多近 | 低 | 高 | 中到低 |
| 时间平面 | 数据是快照、流还是混合 | 离散快照 | 离散快照 | 连续事件流 |
| 权限与门槛平面 | 需要多高代价才能拿到 | 高硬件门槛 | 中高权限门槛 | 相对更低观测门槛 |
| 防御可见平面 | 系统更容易在哪一层发现 | 设备 / 固件 / IOMMU | OS / 进程 / 行为遥测 | 网络栈 / 驱动 / 异常模式 |

这六个平面一旦稳定下来，很多看似杂乱的问题就会自动收敛：

- 为什么抓包能看趋势，却不一定能还原内存里的局部状态
- 为什么内存能看到更高语义对象，却仍然可能不是权威真值
- 为什么 DMA 看似“更底层”，但往往更需要额外解码与时间对齐

### 4.2 三个核心对象

所有未授权程序观测，本质上都围绕三个对象打转：

1. **真值状态**  
   程序或系统真正依赖的状态。有些在客户端，有些只在服务器。

2. **投影状态**  
   真值在某一层留下的可见痕迹，例如内存镜像、传输消息、缓冲区内容。

3. **重建状态**  
   观察者根据投影、规则和时间关系推断出来的高层判断。

防御的关键不是“没有投影”，而是让：

- 高价值真值不长期停在易观测面
- 投影无法稳定反推全部真值
- 重建误差足够大，难以稳定行动

### 4.3 三种观测面最本质的差别

可以把三种路径压成下面这张判断表：

| 维度 | DMA | 进程内存 | 抓包 |
| --- | --- | --- | --- |
| 看见的是 | 原始页级内容 | 运行时状态快照 | 线上的协议事件 |
| 最强项 | 可绕开部分用户态观察边界 | 高语义密度 | 低侵入、连续时序 |
| 最大短板 | 语义距离远 | 容易受布局和权限影响 | 只能看到被发送出来的内容 |
| 最常见误判 | 以为拿到页就等于拿到对象 | 以为客户端内存就是真值 | 以为包流就等于系统全状态 |
| 对防御的启发 | 控制 DMA 可达性 | 降低客户端持有的高价值真值 | 收缩协议泄漏与元数据泄漏 |

### 4.4 一页纸判断模板

以后遇到“某类外挂为什么能做成”这类问题，可以先问下面六个问题：

1. 它依赖的到底是哪一种观测面，还是多面融合？
2. 它看到的是原始数据、对象快照，还是协议事件？
3. 真正关键状态是否仍然长期存在于客户端？
4. 该状态是权威真值，还是渲染/预测/缓存投影？
5. 这条观测链的主要限制是权限、语义距离、时间延迟，还是检测风险？
6. 防御方能否通过改变状态放置位置，直接让这条链失去价值？

## 5. 核心机制 / 主链路 / 因果链

### 5.1 通用主链：从投影到可行动状态

外挂类工具的观测链，从建模上通常都能压成同一条主链：

1. **选择观测面**  
   先判断目标状态更可能暴露在 DMA、进程内存还是网络流上。

2. **拿到原始投影**  
   这一步拿到的还不是“答案”，只是原始字节、对象快照或报文序列。

3. **做结构解码**  
   原始页要映射成对象，内存块要映射成语义字段，报文要映射成协议事件。

4. **做时间对齐**  
   单个样本往往不够。真正有用的是跨时间的变化、关联和趋势。

5. **做高层状态重建**  
   例如把多个底层信号拼成“当前位置”“会话阶段”“服务端裁决结果”“客户端预测值”之类的高层对象。

6. **做反馈修正**  
   如果观察结果和现实反馈不一致，观察者会反向修正自己的解码规则和置信度。

最重要的认知点是：

**外挂类工具真正依赖的不是某个单点读取动作，而是“投影 -> 解码 -> 对齐 -> 重建 -> 修正”这条闭环。**

### 5.2 为什么常常需要多面融合

很多时候，单一观测面并不足以形成稳定高层状态，所以会出现多面融合：

- **抓包补时间。**  
  包流能提供连续事件顺序和网络往返节奏。

- **内存补语义。**  
  内存快照更容易映射到运行时对象和本地状态。

- **DMA补边界外可见性。**  
  DMA 更像带外快照入口，但需要更强的地址和语义映射。

所以，一个更完整的式子是：

**高层状态重建 = 低层投影采样 + 解码规则 + 时间融合 + 交叉校验**

### 5.3 为什么“看到”不等于“控制”

防御建模里很容易犯的一个错误，是把“能观测”直接等同于“能稳定利用”。  
实际上，中间还差四层：

- 观测值是否稳定
- 观测值是否接近权威真值
- 行动延迟是否足够低
- 防御方是否在服务端二次验证

这也是为什么真正有效的防御通常不只做“看不见”，还会做：

- 让客户端只持有局部或预测态
- 让最终裁决在服务端发生
- 让异常动作必须经过权威验证

## 6. 关键 tradeoff 与失败模式

从防御方视角看，这类观测链的核心 tradeoff 是：

**越靠近原始层，语义越远但边界可能更外；越靠近程序语义层，解码越容易但检测与权限风险通常越高。**

最常见的失败模式有八类：

- **把投影误当真值。**  
  客户端缓存、预测值、渲染值和权威值常常不是同一个对象。

- **忽略时间误差。**  
  观测拿到的是过去某一刻的投影，不是永远最新的系统状态。

- **只封单点入口。**  
  封掉一个 API，不代表别的观测面就失效。

- **只做外观混淆。**  
  结构花一点，并不能改变高价值状态仍长期明文驻留这一事实。

- **把网络加密当作总防线。**  
  包流看不懂，不代表内存和 DMA 就没有暴露。

- **把客户端本地计算当作不可见。**  
  只要状态真的留在客户端，它就进入某个观测面。

- **只盯内容，不盯元数据。**  
  即使内容加密，流量方向、节奏、长度和时间关系也可能继续泄漏信息。

- **不做服务器权威验证。**  
  这会让客户端一旦被重建出足够多状态，就能直接把观测转成收益。

## 7. 应用场景

### 7.1 Anti-cheat 威胁建模

这套模型最直接的用途，是把讨论从“又冒出一种工具名”切回“到底是哪类观测面泄漏了高价值状态”。

### 7.2 客户端架构设计

对联网程序、实时系统和游戏客户端来说，这套模型能帮助回答：

- 哪些状态必须留在客户端
- 哪些状态只该以短命投影存在
- 哪些裁决必须收回服务器

### 7.3 协议与遥测设计

抓包只能看到线上投影，但线上投影仍然可能泄漏：

- 会话阶段
- 资源刷新节奏
- 服务端裁决结果
- 动作完成反馈

因此协议设计和防御遥测也必须纳入同一个模型。

## 8. 工业 / 现实世界锚点

### 8.1 Windows Kernel DMA Protection

Microsoft 的 Kernel DMA Protection 说明文档明确把外设 DMA 风险和 IOMMU 隔离联系在一起。  
这说明 DMA 并不是抽象课本对象，而是现代终端安全的真实攻击面和防御面。

### 8.2 Windows 进程对象安全模型

Windows 的 Process Security and Access Rights 文档把进程对象访问权、保护进程限制和 `PROCESS_VM_READ` 等权限边界明确列出来。  
这证明“进程内存可见性”本质上是一条受 OS 安全模型约束的观测面。

### 8.3 libpcap / Npcap / Wireshark 栈

libpcap、Npcap 和 Wireshark 这套生态说明：抓包不是黑科技，而是成熟、标准化的网络观测基础设施。  
也正因为如此，协议设计不能默认“线上没人能看”。

## 9. 当前推荐实践、过时路径与替代

以下“当前实践”相关判断按 `2026-03-24` 组织。

### 9.1 当前更推荐的防御路径

当前更稳的路径通常是：

- **默认客户端是可观测的。**  
  不是假设没人能看，而是先问“看到了也得不到什么关键裁决”。

- **把权威真值收回服务端。**  
  客户端保留显示、预测和交互所需最小集，而不是保留完整裁决权。

- **缩短高价值状态的明文停留时间。**  
  能短命化、派生化、会话化的状态，不要长期明文驻留。

- **协议层做加密和认证，但不把它当唯一防线。**  
  它主要压缩网络投影的语义密度，而不是消灭全部观测面。

- **遥测按“多面关联”设计。**  
  设备、进程、网络三面要能形成统一判断，而不是各自孤立。

### 9.2 已经过时或明显不够的路径

下面这些思路已经明显不够：

- “只要封某个常见 API 就行”
- “只要协议是自定义二进制别人就看不懂”
- “只要地址经常变就没人能重建状态”
- “只要画面上没显示完整真值就安全”

### 9.3 更稳的替代思路

更稳的替代是：

- 从**状态放置**而不是从**表面入口**出发做防御
- 从**投影与真值的距离**而不是从**术语花样**出发做设计
- 从**服务器权威与异常验证**而不是从**客户端保密神话**出发做闭环

## 10. 自测题 / 验证入口

1. 为什么说 DMA、进程内存、抓包不是三种互不相关的技巧，而是三种不同的状态投影面？
2. 为什么抓包看到了完整协议流，仍然不等于看到了程序全部状态？
3. 为什么进程内存看到的是高语义状态，却依然可能不是权威真值？
4. 如果一个系统只收缩了网络投影，却仍在客户端长期明文持有关键状态，会发生什么？
5. 为什么真正有效的防御必须回答“高价值状态到底放在哪里”，而不是只回答“封了哪个入口”？

## 11. 迁移与关联模型

理解这篇文档后，你应该能把模型迁移到：

- EDR 与反篡改中的多面观测
- DRM 与机密资产保护
- 联网游戏或交易终端的客户端可信边界设计
- 逆向分析中的“原始字节 -> 语义对象 -> 行为推断”链路
