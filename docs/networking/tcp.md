# TCP：可靠字节流、连接状态机与反馈控制闭环的统一模型

## 1. 这份文档要帮你学会什么

这篇文档不是为了把 `SYN`、`ACK`、`FIN` 这些标志位背下来，而是要把 TCP 压成一个以后能反复调用的内部模型。

读完后，你应该至少能做到：

- 不再把 TCP 误解成“有连接的 socket”，而是把它看成“可靠字节流契约 + 连接状态机 + 双反馈控制回路”
- 解释为什么 TCP 能把乱序、丢包、重复包压成应用眼里的有序字节流
- 分清“发送方发不动”到底是接收方慢、网络拥塞、应用读不及时，还是连接生命周期本身出了问题
- 预测为什么 `send()` / `recv()` 调用边界不能直接当消息边界
- 把这个模型迁移到应用层 framing、超时设计、长连接治理、排障与协议选型上

如果只记一句话：

**TCP 解决的不是“把包从 A 发到 B”，而是让两个端点在有损、乱序、时延波动的网络上维持一条可恢复、可背压、可收敛的有序字节流关系。**

## 2. 一句话结论 / 问题定义

**TCP 是一种面向连接、可靠、按序交付的字节流传输协议；它通过连接状态机、序列号与确认机制、重传机制，以及接收端与网络两套反馈回路，让应用在不稳定网络上看到一条相对稳定的双向字节流。**

它真正要同时解决的，不只是“可靠送达”这一件事，而是四个问题：

- 两端如何先建立一条不会和旧连接混淆的会话关系
- 数据如何在丢包、乱序、重复的情况下仍被恢复成有序字节流
- 发送方如何同时尊重接收方处理能力和网络路径承载能力
- 连接如何安全结束，避免旧报文污染新会话

所以，TCP 最稳的理解方式不是“头部字段列表”，而是：

- **服务语义是什么**
- **生命周期如何推进**
- **正确性如何维持**
- **发送速率由谁约束**

## 3. 对象边界与相邻概念

### 3.1 TCP 直接处理什么

TCP 直接处理的是：

- 以四元组为基础的端到端连接关系
- 连接建立、数据传输、半关闭与关闭
- 字节流的分段、确认、重传与按序重组
- 接收端背压与网络拥塞反馈
- 基本完整性校验与连接异常重置

### 3.2 TCP 不等于什么

TCP 不等于：

- **消息协议。** TCP 提供字节流，不提供消息边界。
- **安全协议。** 机密性、身份认证、完整性保护通常要靠 TLS 等上层机制补。
- **应用层健康判断。** TCP 连接还活着，不代表业务会话就健康。
- **多流复用协议。** 一个 TCP 连接里的所有应用字节共享同一按序交付语义。
- **低时延保证。** 可靠和按序会带来等待、缓存和重传成本。

### 3.3 最容易混淆的相邻概念

最值得一起看的相邻概念有：

- `socket API`：应用和 TCP 栈交互的编程接口，不等于 TCP 本体
- `TLS`：跑在 TCP 上的安全层，解决的是加密与身份，不是可靠传输
- `QUIC`：在 UDP 上重建部分传输语义，边界与 TCP 很接近，但流模型和演化路径不同
- `HTTP` / `PostgreSQL wire protocol`：这些协议都必须自己定义消息 framing，因为 TCP 不保留消息边界

### 3.4 四组必须先拆开的边界

- **字节流 vs 消息。**
  你写两次 `send()`，对端可能一次 `recv()` 才读出来，也可能拆成多次读出来。

- **连接建立 vs 应用就绪。**
  三次握手完成，只说明传输关系建立了，不说明 TLS、认证、会话初始化等上层动作已经完成。

- **可靠交付 vs 低延迟。**
  TCP 为了保持按序和可靠，会等待丢失数据补齐，这会拖慢后续字节的交付。

- **接收端慢 vs 网络拥塞。**
  两者都会让发送方“发不动”，但一个主要体现为 `rwnd` 压缩，一个主要体现为 `cwnd` 收缩和重传行为。

### 3.5 本文的时间边界

本文在“当前推荐实践、过时路径与替代”部分按 `2026-03-23` 核对。  
这里的“当前实践”主要指互联网与通用工程实践中的默认判断，不代表所有专网、嵌入式或高频交易环境都必须完全一致。

## 4. 核心结构

要把 TCP 建模得足够稳，最好的方式不是背头字段，而是先把它拆成五个彼此耦合的平面。

### 4.1 五个平面

| 平面 | 它解决什么 | 典型对象 |
| --- | --- | --- |
| 服务语义平面 | 应用到底买到了什么契约 | 可靠、按序、字节流、全双工 |
| 生命周期平面 | 连接从无到有、从有到无如何推进 | 4-tuple、3WHS、FIN、RST、TIME-WAIT、状态机 |
| 正确性平面 | 怎么从不可靠网络中恢复连续字节流 | 序列号、ACK、重传、乱序重组、校验和 |
| 反馈控制平面 | 谁来限制发送速度 | `rwnd`、`cwnd`、RTT、RTO、慢启动、拥塞避免 |
| 实现与观测平面 | 应用和运维实际能看到什么 | send/recv buffer、MSS、keepalive、队列长度、重传统计 |

这五个平面缺一不可：

- 只看服务语义，会把 TCP 误解成抽象的“可靠通道”，看不到代价来源
- 只看状态机，会变成背状态图，却解释不了为什么应用读写会卡住
- 只看重传，会忽略发送速率为什么必须被控制
- 只看 `cwnd`，又会忽略很多“发不动”其实是接收端应用层消化不及时

### 4.2 九个必须稳住的结构件

1. **四元组**
   连接不是“两个 IP 的关系”，而是源/目的 IP 与端口共同定义的会话实例。

2. **连接状态机**
   TCP 连接有明确的生命周期状态，不是“连上 / 没连上”两态世界。

3. **序列号空间**
   TCP 不是逐条消息确认，而是围绕字节序列空间确认“哪些字节已经连续收到”。

4. **发送缓存**
   应用写进去的字节先进入发送侧缓冲区，什么时候真正上线、是否重传，不完全由应用立即决定。

5. **接收缓存与按序重组**
   接收端可能先看到后面的段，但只有当前面缺口补齐后，应用才能看到连续字节流。

6. **确认与累计 ACK**
   ACK 不是“我看见一个包”，而是“到某个字节之前的连续空间我都收到了”。

7. **接收窗口 `rwnd`**
   接收端用它告诉发送端“我现在还能吃多少”，这是背压回路。

8. **拥塞窗口 `cwnd`**
   发送端用它约束在途数据量，避免把网络当成无限吞吐的黑洞。

9. **定时器体系**
   RTT 估计、RTO、保活、TIME-WAIT 等时间机制决定 TCP 不只是空间问题，也是时间问题。

### 4.3 一个最值得长期记住的公式

对发送端来说，真正能飞在路上的数据量，不是只看一个窗口，而是：

**在途数据量 <= min(`rwnd`, `cwnd`)**

这句话非常关键，因为它把两类完全不同的问题拆开了：

- `rwnd` 小，说明接收端或其应用消费能力受限
- `cwnd` 小，说明发送端对网络路径的估计变得保守

很多排障如果不先拆这两层，就会把“对端读太慢”错诊成“网络不行”。

### 4.4 一页纸总图

把 TCP 压成最有用的一句话：

**TCP = 字节流契约 + 显式连接状态机 + 依赖序列号/ACK/重传维持正确性 + 同时受接收端与网络两套反馈约束的发送系统。**

## 5. 核心机制 / 主链路 / 因果链

TCP 至少要看四条链：

- 建链
- 传数
- 控速
- 收尾

### 5.1 建链：为什么要三次握手

一个典型 TCP 连接建立链可以压成下面几步：

1. 发起端选择本地四元组并进入主动打开路径。
2. 发起端发送 `SYN`，声明自己的初始序列空间起点。
3. 被动端收到后返回 `SYN+ACK`，一方面确认对方的起点，一方面声明自己的起点。
4. 发起端再发 `ACK`，双方进入可用的数据传输状态。

三次握手真正解决的不是礼节，而是：

- 双方都要知道彼此的初始序列空间
- 双方都要确认对方确实还活着并参与本次连接
- 新连接不能被旧报文误当成当前连接的一部分

所以，握手的本质是：

**先同步序列空间，再建立字节流关系。**

### 5.2 传数：应用写的是字节，网络看到的是段

应用把字节写入 socket 后，TCP 栈会：

1. 把这些字节暂存到发送缓冲区
2. 根据 MSS、拥塞状态、接收窗口等因素决定如何切分成段
3. 为每段分配覆盖其字节区间的序列号
4. 发到网络上，等待 ACK 或更强的丢失信号

接收端则会：

1. 校验段是否合法
2. 根据序列号放入接收缓冲区
3. 对连续收到的最高字节位置发送累计 ACK
4. 只有当缺口被补齐后，才把连续字节流交给应用

这里最容易踩坑的是：

**应用层写入边界和 TCP 段边界没有稳定对应关系。**

TCP 可以为了效率或控制目的合并、拆分、重传和重排发送单元；应用看到的只是一个连续字节流。

### 5.3 纠错：TCP 不是“不丢”，而是“丢了还能补”

TCP 的可靠性不是靠假设网络不出错，而是靠三件事组合：

- **序列号**
  让每个字节在流里有唯一位置。

- **ACK**
  让发送端知道接收端已经连续收到哪里。

- **重传**
  让缺失部分在检测到超时或丢失迹象后重新进入传输。

这意味着 TCP 的正确性建立在“可恢复”上，而不是“永不失败”上。  
如果路径持续严重拥塞、两端一直收不到确认、连接被复位，TCP 仍然会失败，只是失败前会尽量自恢复。

### 5.4 控速：TCP 里同时有两套反馈回路

TCP 最容易被低估的部分，是它不只有一套“窗口”，而是至少两套约束。

| 回路 | 它在约束什么 | 核心信号 |
| --- | --- | --- |
| 接收端背压回路 | 对端还能接收多少 | `rwnd`、应用读取速度、接收缓冲区占用 |
| 网络拥塞回路 | 路径还能承受多少在途数据 | `cwnd`、RTT、丢包、ECN、重传模式 |

这两套回路会共同决定发送行为。

典型因果链是：

1. 如果接收端应用读得慢，接收缓冲区被填满
2. 接收端通告的 `rwnd` 下降
3. 发送端即使网络很空，也要收缩发送

另一条链则是：

1. 如果路径拥塞或丢包增加
2. 发送端从 ACK、RTT、重传中感知风险
3. `cwnd` 收缩，发送速率下降
4. 随着网络恢复，再逐步探测更高发送速率

所以，TCP 不是单纯的“可靠传输”，它本质上还是一个闭环控制系统。

### 5.5 收尾：为什么关闭不是“啪一下断掉”

TCP 关闭连接至少要区分三种情况：

- **正常关闭**
  一端发送 `FIN`，表示“我没有更多字节要发了”，但对向还可以继续发送。

- **半关闭**
  一侧写方向关闭，读方向仍然保留。这对某些请求完成后继续收剩余数据的模式很重要。

- **异常终止**
  用 `RST` 快速打断连接，通常意味着状态不一致、端口未监听或应用主动放弃会话。

最关键的生命周期尾部状态是 `TIME-WAIT`。  
它的重要性不在于“多等一会儿”，而在于：

- 避免旧连接残留段干扰后续重用同一四元组的新连接
- 允许最终 ACK 等收尾报文有时间在网络中自然消散

很多人把 `TIME-WAIT` 当成无意义负担，但它其实是在替连接边界擦屁股。

## 6. 关键 tradeoff 与失败模式

### 6.1 五个核心 tradeoff

| 设计点 | 你得到什么 | 你付出什么 |
| --- | --- | --- |
| 可靠 + 按序 | 应用逻辑更简单 | 丢一段会拖住后续字节交付，形成 HOL 阻塞 |
| 显式建链 | 明确连接边界与序列同步 | 额外握手时延与状态维护成本 |
| 内建拥塞控制 | 对网络更友好，长期更稳 | 吞吐与时延会随路径状态动态波动 |
| 接收端背压 | 不容易直接把慢接收者压垮 | 应用读不及时会反向限制发送端 |
| 平滑字节流接口 | 通用性高，适合很多协议 | 上层必须自己做 framing 和消息边界管理 |

### 6.2 最常见的失败模式

- **把 `send()` / `recv()` 边界当消息边界**
  这是最经典的应用层 bug。TCP 从未承诺这件事。

- **把“连接成功”当“业务可用”**
  TLS 握手、认证、应用初始化都还没算进去。

- **把所有传输卡顿都归因于网络**
  很多卡顿其实是接收端应用没及时读数据，导致 `rwnd` 收缩。

- **把 TCP 当实时媒体协议**
  丢失后的重传和按序语义，会让过期数据挡住新数据。

- **把 keepalive 当业务 liveness 机制**
  TCP keepalive 的语义、频率和故障检测时效，通常都不等于你的业务需要。

- **用 TCP 上再叠一层“自定义可靠传输”**
  很容易与 TCP 自己的重传和缓冲行为打架，形成双重等待与双重队列。

### 6.3 三个高频误诊

1. **“网络没问题，因为能 ping 通。”**
   Ping 能通，只说明 ICMP 或路径基本可达，不能说明 TCP 会话和应用协议一定健康。

2. **“吞吐低，是因为带宽不够。”**
   也可能是窗口太小、RTT 太高、丢包触发拥塞控制，或者接收端读取跟不上。

3. **“包已经到了，对端为什么没读到？”**
   TCP 可能已经收到后续字节，但前面有缺口，所以应用仍看不到后面的数据。

## 7. 应用场景

### 7.1 关系型数据库与状态型请求流

像 PostgreSQL、MySQL 这类数据库协议，需要：

- 有状态会话
- 请求和响应严格关联
- 数据不能乱序或随便丢

它们通常把应用消息 framing 建在 TCP 字节流之上，而不是要求 TCP 直接替它保存消息边界。

### 7.2 Web、代理与控制平面

HTTP/1.1、HTTP/2、反向代理、服务网关、大量内部 RPC 都大量依赖 TCP，因为它们更看重：

- 广泛兼容性
- 成熟的 TLS 叠加方式
- 对有损网络下正确性的稳定抽象

在这些场景里，TCP 的价值不只是“能传”，而是让复杂应用协议不必自己重写一整套可靠传输控制面。

### 7.3 SSH、复制与文件传输

像 SSH 会话、数据库复制、文件传输、日志流式复制这类场景，最怕的是：

- 字节丢失
- 顺序错乱
- 一边慢了另一边完全不知情

TCP 的可靠、按序、可背压特性在这里非常匹配。

## 8. 工业 / 现实世界锚点

### 8.1 PostgreSQL：TCP 不负责消息 framing，所以数据库协议必须自己做

PostgreSQL 的前后端协议跑在 TCP 上，但它并不依赖“一个请求就是一个包”。  
它通过长度字段和协议状态自己定义消息边界，这正好说明：

- TCP 给它的是可靠字节流
- 真正的业务消息结构必须由上层自己定义

这是理解“TCP 不是消息协议”的非常典型锚点。

### 8.2 HTTP/2 over TCP：单连接多流不等于没有传输级 HOL

HTTP/2 在应用层做了多路复用，但它仍然通常承载在单条 TCP 连接之上。  
这意味着：

- 应用层确实拿到了多 stream 语义
- 但底层一旦出现 TCP 丢包，整个连接的后续字节推进仍会受影响

这也是后来 HTTP/3 / QUIC 在互联网场景中获得动力的重要背景之一。

### 8.3 OpenSSH：当你真正需要的是一条可信赖、长期存在的控制流

SSH 的核心不是“发消息很快”，而是：

- 字节不能丢
- 顺序不能乱
- 会话和背压要稳定

这类控制平面协议与 TCP 的契合度极高，因此几十年都稳定建立在 TCP 之上。

## 9. 当前推荐实践、过时路径与替代

以下“当前推荐实践”于 `2026-03-23` 核对，面向通用互联网与通用工程系统。

- **当前推荐：** 当你的应用真正需要可靠、按序、面向字节流的语义时，默认先选 TCP。  
  **原因：** 这类需求如果自己在 UDP 上重写，几乎一定会重新发明重传、拥塞控制、会话状态与异常恢复。  
  **什么时候尤其合适：** 数据库协议、控制平面、文件传输、传统 Web/TLS 栈、内部 RPC。

- **当前推荐：** 在 TCP 之上显式做应用层 framing，而不是偷用调用边界。  
  **原因：** TCP 对应用暴露的是连续字节流，不保留消息边界。  
  **典型做法：** 长度前缀、分隔符、定长头 + 变长体、协议状态机。

- **当前推荐：** 在不可信网络上，把 TLS 看作 TCP 的常规配套，而不是可选装饰。  
  **原因：** TCP 解决的是传输正确性，不解决身份、机密性和抗篡改。

- **过时路径：** 把 TCP 当“天然消息通道”。  
  **为什么旧：** 这会直接催生粘包 / 拆包类 bug，本质上是把字节流误当记录流。  
  **现在更推荐：** 明确定义 framing，并把“读到完整一条消息”的条件写进协议层。

- **过时路径：** 用 TCP keepalive 代替业务层心跳和超时模型。  
  **为什么旧：** keepalive 主要是传输层的闲置连接探测机制，时间尺度通常不匹配业务 SLA。  
  **现在更推荐：** 在应用层设计明确的 deadline、heartbeat、幂等重试和断线恢复逻辑。

- **替代路径：** 当你需要多流复用、连接迁移、更细粒度的丢失隔离，且能够接受新传输栈时，优先比较 QUIC，而不是直接跳到“那就裸 UDP”。  
  **原因：** QUIC 不是抛弃传输控制，而是把很多 TCP 的 transport 职责搬到了 UDP 之上的新实现中。  
  **什么时候选它：** 面向互联网的浏览器流量、连接迁移重要、单连接内多逻辑流时延耦合很敏感的场景。

## 11. 迁移与关联模型

学会这篇文档后，你可以把模型迁移到：

- **应用协议设计**
  先问自己买到的是字节流还是消息边界，再决定 framing、重试与幂等。

- **TCP 与 QUIC 选型**
  先问你真正缺的是“可靠性”还是“连接级 HOL 隔离、迁移与用户态演化能力”。

- **网络排障**
  看到“发不动”时，先分 `rwnd`、`cwnd`、RTO、应用读写节奏，而不是只盯带宽。

- **长连接治理**
  先分传输连接存活、TLS 会话健康、业务会话健康三个层次，不要混成一层。

最值得保留的迁移句式是：

**先问你的协议真正买的是字节流契约，还是消息契约；再问它卡住时，是生命周期问题、正确性恢复问题，还是反馈控制问题。**
